# ReBLOG 开发日记

> 本文档记录 ReBLOG 个人博客内容管理系统的完整开发过程，供毕业论文参考。

## 一、项目概述

**项目名称**：ReBLOG — 个人博客内容管理系统

**项目定位**：面向个人用户的轻量级博客系统，支持文章发布、图库管理、日记记录、评论互动、文件管理等功能。采用前后端分离架构，包含管理后台（ReBLOG-admin）与后端 API 服务（ReBLOG-server），后续将扩展面向访客的公开展示页面（ReBLOG-web）。

**开发者**：REDEYES

**开发周期**：2026 年 4 月 — 2026 年 7 月（持续迭代中）

在项目启动阶段，我对当前主流的前后端技术栈进行了调研与对比。考虑到项目为个人独立开发，需要在开发效率、学习成本、社区生态之间取得平衡。

### 2.4 选型反思

在开发过程中，以下选型决策值得记录：

1. **直接 SQL 拼接 vs ORM**：初期选择了手写 SQL（通过 mysql2），而非使用 Sequelize/TypeORM。好处是 SQL 控制力强、调试直观；缺点是在复杂查询（多条件分页、模糊搜索）时代码重复度高，且存在 SQL 注入风险（如搜索关键字的 LIKE 拼接未做参数化处理）。如果项目规模扩大，应考虑引入 Knex 等查询构建器。

2. **单一 controller 文件**：所有业务逻辑集中在 `controller/serve.js`（540 行），随着接口数量增长，文件变得臃肿。更合理的做法是按模块拆分（articleController、commentController、diaryController 等）。

3. **密码明文存储**：注册接口将密码直接存入数据库，未做 bcrypt 哈希处理。这是安全性上的一个遗留问题。

### 3.2 路由设计

采用 POST-only 的 API 设计（所有请求通过 POST 方法，参数放在 JSON Body 中），主要考虑：
- 统一请求格式，前端无需区分 GET/POST 参数位置
- Token 统一从 `req.body.token` 获取
- 避免 URL 中暴露查询参数

API 共 31 个端点，按功能模块分类：

| 模块 | 端点 | 功能 |
|---|---|---|
| 认证 | `/register`, `/signin`, `/isRegister` | 管理员注册、登录、状态检查 |
| 文章 | `/aritcle`, `/createArticle`, `/gainArticle`, `/updateArticle`, `/deleteAritcle`, `/changeAritcleState`, `/articleState` | 文章 CRUD、发布状态管理、状态统计 |
| 评论 | `/comment`, `/commentIsread`, `/deleteComment` | 评论列表、已读标记、删除 |
| 私信 | `/message`, `/noreadMessage`, `/deleteMessage` | 私信管理 |
| 日记 | `/diary`, `/createDiary`, `/deleteDiary` | 日记 CRUD |
| 文件 | `/file`, `/upload`, `/removeFile`, `/deleteFile` | 文件上传、分组移动、删除 |
| 分组 | `/subset`, `/addSubset`, `/updateSubset`, `/deleteSubset` | 分类管理 |
| 标签 | `/label`, `/addLabel`, `/deleteLabel` | 标签管理 |
| 总览 | `/overview` | 仪表盘数据统计 |

### 3.3 数据库设计

![数据库 ER 图](未绘制)

共设计 9 张表：

- **users** — 管理员账户（id, name, password）
- **article** — 文章与图库统一存储，通过 `classify` 字段区分（0=文章, 1=图库），通过 `state` 控制发布状态（0=未发布, 1=已发布）
- **diary** — 个人日记（title, content, weather_id, picture）
- **comment** — 文章评论（user_id, user_name, article_id, content, isread）
- **message** — 私信/留言（user_id, user_name, content, isread）
- **file** — 上传文件元数据（url, file_name, format, subset_id）
- **subset** — 分组/分类（subset_name, classify）
- **label** — 文章标签（label_name）
- **praise** — 文章点赞（user_id, article_id）

设计要点：
- article 和 gallery 共用同一张表，通过 `classify` 字段区分。这样避免了重复的表结构和业务逻辑。
- 标签使用逗号分隔字符串存储在 article 的 `label` 字段中，而非多对多关联表。在个人博客规模下，这种简化是可接受的。
- 分组（subset）同时服务于文章、图库、文件三种资源，通过 `classify` 区分所属模块。

## 五、技术难点与解决方案

### 5.1 Token 中间件的异步陷阱

**问题描述**：`jwt.verify()` 是异步方法，通过回调返回结果。但 `verifyToken()` 函数在调用 `jwt.verify()` 后同步 `return payload`，此时回调尚未执行，payload 始终为 undefined。导致中间件认为所有 Token 均无效。

```javascript
// 有问题的代码
exports.verifyToken = (e) => {
  let payload
  jwt.verify(e, secret, (error, decoded) => {
    if (error) { payload = 0 }
    else { payload = 1 }
  })
  return payload  // 始终返回 undefined
}
```

**解决方案**：改为同步调用 `jwt.verify(e, secret)`（不带回调时返回 decoded 对象），或使用 `util.promisify` 包装为 Promise，在中间件中使用 async/await。

**教训**：使用第三方库的异步 API 时，必须理解其回调/Promise 语义。TypeScript 的类型检查也无法捕获这种逻辑错误（payload 的类型是隐式 any）。

### 5.2 文章与图库的统一数据模型

**设计决策**：article 和 gallery 共用同一张表，通过 `classify` 字段区分（0=文章, 1=图库）。

**优点**：
- 避免维护两套几乎相同的表结构和 CRUD 代码
- 标签、分组、评论、点赞等关联功能完全复用
- 数据库迁移和备份更简单

**缺点**：
- 查询时始终需要 `classify` 条件过滤
- article 的 `content`（富文本 HTML）和 gallery 的 `content`（空格分隔的图片 JSON）语义不同，增加了前端解析的复杂度
- 如果未来两种内容类型的差异变大（如文章需要 SEO 元数据、图库需要 EXIF 信息），重构成本高

**反思**：在内容类型差异较小时，统一模型是合理的选择。如果差异持续扩大，可以考虑使用单表继承（STI）模式或 JSON 扩展字段。

### 5.3 多条件分页查询的 SQL 拼接

**问题描述**：文章列表支持 5 个维度的筛选（发布状态、分组、搜索关键字、文章/图库分类、标签），每种组合对应不同的 SQL WHERE 子句。

**当前实现**：在 `db_model.js` 中使用 if-else 链拼接 SQL：

```javascript
if (serchTerm) {
  _sql = `... LIKE "%${serchTerm}%" ...`
} else if (subsetId > -1) {
  _sql = `... subset_id="${subsetId}" ...`
} else if (state > -1) {
  _sql = `... state="${state}" ...`
} else {
  _sql = `...`
}
```

**问题**：
1. **SQL 注入风险**：`serchTerm` 直接拼入 LIKE 子句，未做参数化处理。恶意输入可构造 SQL 注入攻击。
2. **条件互斥**：if-else 链意味着各筛选条件是互斥的，无法同时使用。这是设计上的限制而非 bug（前端 UI 也是互斥的）。
3. **代码重复**：`getArticlePage` 和 `articleCount` 的 WHERE 逻辑几乎相同，维护时容易遗漏同步。

**改进方向**：使用 Knex 查询构建器动态组合 WHERE 条件，或至少使用参数化查询（`?` 占位符）。

### 5.4 前端状态筛选的交互逻辑

**问题描述**：文章列表页同时有"发布状态筛选"（全部/已发布/未发布）和"分组筛选"（全部分组/具体分组）。两者的选项不是独立组合的——选择具体分组后，状态筛选应自动重置为"全部"，反之亦然。

**实现方案**：
- 使用 Vue 的 `ref` 分别管理 `nowState` 和 `nowSubset`
- 在分组选择事件中：清除 `nowState`（设为 -1），设置 `nowSubset`
- 在状态选择事件中：清除 `nowSubset`（设为 -1），设置 `nowState`
- 搜索时：隐藏全部分组/状态筛选，只显示搜索结果

**反思**：这种交互模式在 CMS 后台中很常见，但容易让用户困惑为什么之前的筛选被清除了。更好的做法是允许组合筛选，或在 UI 上明确提示当前筛选状态。

## 七、经验总结

### 7.1 做得好的地方

1. **前后端分离**：清晰的架构分层使得前后端可以独立开发和调试，互不阻塞。
2. **组件化开发**：Vue 单文件组件的粒度控制得当，article/gallery/diary 的卡片组件可独立复用。
3. **TypeScript 引入**：接口定义（interface.ts）为数据结构提供了文档和约束，在重构时避免了大量字段错误。
4. **Pinia 状态管理**：相比 Vuex，Pinia 的 API 更简洁，TypeScript 推导更完整，开发体验显著提升。

### 7.2 需要改进的地方

1. **后端代码组织**：controller 单文件过大，应按模块拆分。model 层 SQL 拼接应升级为查询构建器。
2. **安全性**：密码明文存储、SQL 注入风险、CORS 允许所有来源（`Access-Control-Allow-Origin: *`）——这些问题需要在正式上线前修复。
3. **测试缺失**：项目中没有任何自动化测试（单元测试、集成测试、E2E 测试），所有验证依赖手动操作。
4. **日志与监控**：没有请求日志、错误追踪、性能监控，线上问题排查困难。
5. **API 版本管理**：没有 API 版本前缀（如 `/api/v1/`），后续 API 变更可能破坏兼容性。

### 7.3 对后续开发者的建议

1. 在项目初期就引入 ESLint + Prettier，保持代码风格一致。
2. API 设计尽量遵循 RESTful 规范（使用 GET 获取资源、使用语义化的 URL），不要因为方便而全部使用 POST。
3. 尽早建立 CI/CD 流程（代码检查 → 测试 → 自动部署），避免手动操作带来的风险。
4. 使用 Docker 容器化部署，减少环境差异导致的问题。
